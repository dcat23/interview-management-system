## Topics

### `session.status.changed`

Published whenever a session status transitions â€” whether triggered manually by a user or automatically by the background job.

**Producers:**
- `apps/api` â€” on `PATCH /sessions/:id/status`
- `services/background-job` â€” on auto `scheduled â†’ in_review` transition

**Consumers:**
- `services/background-job` â€” updates `status_history`, triggers notification events

**Partitioning:** by `process_id` (ensures all events for a process arrive in order)

**Partitions:** 6

---

#### Event: `SessionStatusChanged`

```json
{
  "event_type": "SessionStatusChanged",
  "event_id": "uuid",
  "occurred_at": "2024-02-15T15:00:00Z",
  "schema_version": "1.0",
  "payload": {
    "session_id": "uuid",
    "process_id": "uuid",
    "candidate_id": "uuid",
    "end_client_id": "uuid",
    "round": "1st round",
    "from_status": "scheduled",
    "to_status": "in_review",
    "changed_by": "uuid | null",
    "change_source": "manual | background_job",
    "scheduled_at": "2024-02-15T14:00:00Z"
  }
}
```

**Field notes:**
- `event_id`: UUIDv4 generated by producer. Used by consumer for idempotency check.
- `changed_by`: `null` when `change_source` is `background_job`.
- `from_status`: `null` on initial session creation event.
- `schema_version`: consumers must check this field and handle unknown versions gracefully.

**Status transition events emitted:**

| Transition | `change_source` | Triggered by |
|---|---|---|
| `null â†’ scheduled` | `manual` | Marketer / admin creates session |
| `scheduled â†’ in_review` | `background_job` | Background job after `scheduled_at` passes |
| `scheduled â†’ in_review` | `manual` | Marketer or supporter |
| `scheduled â†’ cancelled` | `manual` | Marketer or admin |
| `scheduled â†’ no_show` | `manual` | Supporter, marketer, or admin |
| `in_review â†’ passed` | `manual` | Supporter, marketer, or admin |
| `in_review â†’ rejected` | `manual` | Supporter, marketer, or admin |
| `in_review â†’ no_show` | `manual` | Supporter, marketer, or admin |

### `notifications.send`

Published when a notification should be sent to a user. Decouples notification delivery from the API and background job.

**Producers:**
- `services/background-job` â€” consumes `session.status.changed` and publishes notification events based on transition type

**Consumers:**
- Notification worker (within `services/background-job` as a separate consumer group) â€” sends email via AWS SES

**Partitions:** 3

**Partitioning:** by `recipient_id`

---

#### Event: `NotificationRequested`

```json
{
  "event_type": "NotificationRequested",
  "event_id": "uuid",
  "occurred_at": "2024-02-15T15:00:00Z",
  "schema_version": "1.0",
  "payload": {
    "notification_type": "SESSION_SCHEDULED | SESSION_IN_REVIEW | SESSION_OUTCOME | FEEDBACK_SUBMITTED",
    "recipient_id": "uuid",
    "recipient_email": "string",
    "recipient_role": "candidate | marketer | supporter | admin",
    "template_data": {
      "candidate_name": "string",
      "end_client_name": "string",
      "round": "string",
      "scheduled_at": "2024-02-15T14:00:00Z",
      "mode": "string",
      "status": "string"
    }
  }
}
```

**Notification types and recipients:**

| `notification_type` | Trigger | Recipients |
|---|---|---|
| `SESSION_SCHEDULED` | Session created (`null â†’ scheduled`) | Candidate, supporter |
| `SESSION_IN_REVIEW` | Transition to `in_review` | Admin, marketer |
| `SESSION_OUTCOME` | Transition to `passed`, `rejected`, or `no_show` | Admin, marketer |
| `FEEDBACK_SUBMITTED` | Supporter submits feedback (`is_submitted = true`) | Admin |

## Idempotency

All consumers must be idempotent. The `event_id` field is used to detect and discard duplicate deliveries.

**Implementation pattern (background job):**

```java
// Before processing, check if event_id has been seen
if (processedEventRepository.existsByEventId(event.getEventId())) {
    log.info("Duplicate event ignored: {}", event.getEventId());
    return;
}
// Process event
// ...
// Mark as processed
processedEventRepository.save(new ProcessedEvent(event.getEventId()));
```

The `processed_events` table:

```sql
CREATE TABLE processed_events (
  event_id    uuid PRIMARY KEY,
  processed_at timestamptz NOT NULL DEFAULT now()
);

-- Prune records older than 48 hours via scheduled cleanup
CREATE INDEX idx_processed_events_at ON processed_events(processed_at);
```

## Dead letter queue

Failed messages (after 3 retry attempts) are routed to a dead letter topic:

- `session.status.changed.dlq`
- `notifications.send.dlq`

DLQ messages are monitored via a CloudWatch alarm on topic consumer lag. On-call team investigates manually. Re-processing is done by replaying the DLQ topic with a dedicated consumer.

---

## Schema evolution

- Schema changes are **additive only** (new optional fields). No field removals or type changes on existing fields.
- `schema_version` is incremented on any structural change.
- Consumers must ignore unknown fields (Jackson `FAIL_ON_UNKNOWN_PROPERTIES = false`).
- Breaking changes require a new topic (`session.status.changed.v2`) with a migration period during which both topics are produced to.
